# 业务规则记录 (Business Rules)

## 1. 订单状态跳转规则 (Order Status Transitions)
订单状态（Order Status）代表整个销售订单的生命周期。

### 1.1 状态定义与前置条件
| 状态代码 | 状态名称 | 业务含义 | 前置条件 (至少包含) |
| :--- | :--- | :--- | :--- |
| **S00** | 意向报价中 | 订单刚创建，还没有设计方案跟报价单 | 无任何已发布内容 |
| **S01** | 意向沟通中 | 客户收到设计报价单，还未有任何反馈 | 1个已发布方案 + 1个已发布报价单 |
| **S02** | 订单深化中 | 方案深化设计、详细图纸绘制、精准报价阶段 | 1个已发布方案 + 1个已发布报价单 |
| **S02-01** | 提案设计中 | 方案深化设计、详细图纸绘制 | 1个已发布方案 + 1个已发布报价单 |
| **S02-02** | 订购报价中 | 精准报价阶段 | 1个已发布方案 + 2个已发布报价单 |
| **S03** | 订购确认中 | 向客户提交最终深化方案和精准报价，等待客户确认 | 1个已发布方案 + 2个已发布报价单 |
| **S04** | 客户已婉拒 | 客户觉得报价或者方案不满意，决定不成交 | 1个已发布方案 |
| **S05** | 客户决策中 | S01到期自动跳转，客户迟迟未决策 | 1个已发布方案 + 2个已发布报价单 |
| **S06-01** | 交付设计中 | 正式进入交付阶段，进行施工图、节点图等详细设计 | 1个已发布方案 + 2个已发布报价单 |
| **S06-02** | 方案汇报中 | 向客户汇报最终施工方案及图纸 | 1个已发布方案 + 2个已发布报价单 |
| **S06-03** | 交付备货中 | 方案通过后，进行材料采购、工人调度等准备工作 | 1个已发布方案 + 2个已发布报价单 |
| **S06-04** | 交付施工中 | 正式进场施工，按节点推进 | 1个已发布方案 + 2个已发布报价单 |
| **S06-05** | 交付内审中 | 施工完成，现场进行内部审核验收 | 1个已发布方案 + 2个已发布报价单 |
| **S07** | 订单验收中 | 施工完成，提交竣工验收及最终结算单，等待确认 | 1个已发布方案 + 2个已发布报价单 + 1个结算单 |
| **S08** | 订单终止中 | 订单因各种原因提前终止 | 1个已发布方案 + 2个已发布报价单 |
| **S09** | 订单整改中 | 客户对验收或结算有异议，进行内部整改 | 1个已发布方案 + 2个已发布报价单 + 1个结算单 |
| **S10** | 订单维保中 | 交付后，客户发起售后维修或保养申请 | 1个已发布方案 + 2个已发布报价单 + 1个结算单 |
| **S11** | 订单已交付 | 验收通过，结清尾款，订单正常完结 | 1个已发布方案 + 2个已发布报价单 + 1个结算单 |
| **S12** | 订单已结束 | 订单生命周期完全结束 | 1个已发布方案 + 2个已发布报价单 + 1个结算单 |
| **S13** | 订单休眠中 | 订单因外部原因暂时停工 | 1个已发布方案 + 2个已发布报价单 |

### 1.2 Builder端状态跳转逻辑
在Builder端，仅涉及以下状态的主动跳转：
- **S00 -> S01**: 发布**第一个**报价单时触发。
- **S02 -> S03**: 发布**第二个**报价单时触发。
- **注意**: 发布设计方案（Scheme）**不会**影响订单状态，不会触发任何状态跳转。
- **其他状态**: 其余状态（如 S04, S05, S06-X 等）由服务端控制，Builder端仅读取并显示。

## 2. 方案设计版本状态规则 (Scheme Version Status)
方案状态（Version Status）代表单个设计方案版本的状态。

| 状态代码 | 状态名称 | 描述 | 位置 |
| :--- | :--- | :--- | :--- |
| `draft` | 草稿版本 | 内部创建，未点击发布 | 当前活跃版本区 |
| `published_unread` | 已发布(未读) | 内部点击发布，客户尚未开始审阅 | 当前活跃版本区 / 历史列表 |
| `reviewing` | 客户审阅中 | 客户在终端点击“开始审阅” | 当前活跃版本区 / 历史列表 |
| `reviewed` | 客户已完成审阅 | 客户点击“完成审阅 / 满意并提交” | 当前活跃版本区 / 历史列表 |
| `historical` | 历史版本 | 仅作为状态占位，实际显示时保留原状态名称 | 历史与已发布列表 |

### 5.1 详细分布

2. **订购期**: S02-订单深化中, S03-订购确认中
3. **交付期**: S06-订单交付中, S13-订单休眠中
4. **验收期**: S07-订单验收中, S09-订单整改中, S08-订单终止中
5. **维保期**: S11-订单已交付, S10-订单维保中
6. **终止状态**: S04-客户已婉拒, S12-订单已结束 (颜色同意向期)

### 3.1 唯一草稿约束 (Unique Draft Constraint)
- 整个订单下**只能存在一个**处于 `draft` 状态的方案。
- 如果当前已存在 `draft`，则**不允许**新建方案。用户必须先发布当前草稿。

### 3.2 版本号递增规则 (Version Numbering)
- 版本号采用纯数字递增：1, 2, 3...
- UI 显示为“内部版本 X”。
- 全局递增，不允许删除版本。
- 每次新建方案时，版本号 = 当前最大版本号 + 1。

### 3.3 方案名称编辑 (Editable Name)
- 仅当方案状态为 `draft` 时允许编辑名称。

### 3.4 复制与继承逻辑 (Copy & Inheritance)
- 新建方案时，自动基于当前订单的**最新版本**生成副本。
- 副本将继承原版本的所有信息，包括：
    - 设计图页面
    - EPC设计注释（标注）
    - 客户反馈参考（评论）
- 新建成功后停留在当前页面，并弹出提示告知用户已继承相关数据。

### 3.5 历史列表显示 (History List Display)
- 被挤入“历史与已发布版本”列表的方案，**保持其原有的状态名称**。
- 历史列表中所有的状态胶囊统一显示为**灰色**。

### 3.6 报价单与结算单状态规则 (Quotation & Settlement Status)
报价单（Quotation）与结算单（Settlement）采用一致的状态流转逻辑。

#### 3.6.1 活跃版本逻辑
- 报价单/结算单同样具有“活跃状态”概念。
- 当新的单据（新版本）发布后，旧的单据自动进入“历史与已发布版本”列表。

#### 3.6.2 状态定义
| 状态名称 | 描述 | 触发动作 (驱动变化) |
| :--- | :--- | :--- |
| **未查看未签字** | 初始状态，单据已生成并发布 | Builder端点击“发布” |
| **已查看未签字** | 客户已阅读单据 | 客户打开单据详情页 |
| **已查看已反馈** | 客户已提交修改意见 | 客户点击“希望调整方案”并提交反馈 |
| **已查看已签字** | 客户已确认单据 | 客户点击“确认单据”并完成签名 |

#### 3.6.3 状态流转路径
1. **未查看未签字** --(客户查看)--> **已查看未签字**
2. **已查看未签字** --(客户签字)--> **已查看已签字**
3. **已查看未签字** --(客户反馈)--> **已查看已反馈**
4. **已查看已反馈** --(Builder端发布新版)--> **未查看未签字** (新单据生成，旧单据归档)

### 3.7 视图重置 (View Reset)
- 每次进入 `/overview` 页面，视图自动重置到最顶部。
